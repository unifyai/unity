## File Summaries

Here's a summary of the purpose of the key files within the `unity` project:

**Root Level & Core Orchestration**

1.  **`make_call.py`**:
    *   **Purpose**: Main entry point for the **voice agent** functionality if *not* using the `off_the_shelf` browser automation. Orchestrates LiveKit connection, voice I/O, and interaction with backend agent logic via Redis channels.
    *   **Functionality**:
        *   Sets up logging and environment variables.
        *   Initializes `BusManager` (conceptually, though Redis is now the bus).
        *   `VoiceAssistant(Agent)`: Handles LiveKit STT. After STT, it formats conversation history and prepares to publish the latest transcript.
        *   It uses tools (`is_task_running`, `create_task`, etc.) to interact with a Redis-backed state (`state.py`) and potentially trigger browser tasks via the `transcript` Redis channel (by calling `create_task`).
        *   `_speech_dispatcher` (conceptual, now part of `make_call.py`'s main async logic): Listens to the `task_completion` Redis channel. When a message arrives (signaling task completion from `Planner` or `BrowserAssistant` in `off_the_shelf.py`), it interrupts any ongoing TTS and makes the agent speak a confirmation.
        *   `entrypoint`: Main async function for LiveKit. Connects to LiveKit, starts `BusManager` components (which would now be separate processes like `TaskManager`, `Planner`, `Controller` *or* `BrowserAssistant`/`BrowserController` from `off_the_shelf.py`), configures `AgentSession` (STT, LLM, TTS), starts `VoiceAssistant`, and the speech dispatcher logic.

2.  **`new_make_call.py`**:
    *   **Purpose**: An alternative or refactored version of `make_call.py` that seems to integrate more directly with the `browser_use` library (via `off_the_shelf.py`'s components or similar direct usage).
    *   **Functionality**:
        *   The `VoiceAssistant` here directly manages a `BrowserAgent` instance from the `browser_use` library.
        *   It creates a separate asyncio event loop (`_browser_loop`) within a dedicated thread (`_browser_thread`) to run browser tasks non-blockingly.
        *   The `create_task` tool method now directly calls `self._browser_agent.add_new_task()` with the conversation history and then schedules `self.browser_run()` on the `_browser_loop`.
        *   `notify_task_completed_wrapped` is a mechanism to signal back from the browser thread to the main LiveKit event loop when a browser task finishes. This is crucial for telling the user the task is done.

3.  **`start_chat.py`**:
    *   **Purpose**: Provides a pure-text, asynchronous console chat interface, likely for testing the core agent logic without voice.
    *   **Functionality**:
        *   Uses `asyncio.Queue` for internal message passing between user input, agent logic, and display.
        *   `call_llm`: Placeholder for actual LLM interaction for chat responses.
        *   Publishes full chat history to `bus_manager.transcript_q` (which would map to the `transcript` Redis channel in the current architecture).
        *   Listens to `bus_manager.task_completion_q` (the `task_completion` Redis channel) to announce task completions.
        *   Starts `BusManager` and its components (e.g., `TaskManager`, `Planner`, `Controller`, or the `off_the_shelf` components).

4.  **`new_start_chat.py`**:
    *   **Purpose**: A terminal-based chat application using the `textual` library for a richer Text User Interface (TUI).
    *   **Functionality**:
        *   Uses `textual` widgets for input and message display.
        *   Integrates with an `AsyncUnify` client (e.g., `gpt-4o@openai`) for LLM responses.
        *   Manages message history and displays AI typing indicators.
        *   Uses `textual.work` for background LLM calls (non-blocking UI).
        *   `UserAgentManager`: A conceptual class to manage event flow and agent responses, though the TUI directly calls the LLM here.
        *   Has a basic event queue system (internal to `ChatApp`) for handling user inputs and typing events.

5.  **`test_controller.py`** (in root):
    *   **Purpose**: A Tkinter-based GUI application to directly interact with and test the `BrowserWorker` and parts of the `Controller` logic (specifically `agent.py`).
    *   **Functionality**:
        *   Sets up a `ControlPanel` GUI (from `controller.gui`).
        *   Starts a `BrowserWorker` thread.
        *   Uses a Python `queue.Queue` (`gui_to_browser_queue`) and a dedicated thread (`_cmd_forwarder`) to publish commands entered in the GUI to the `browser_command` Redis channel.
        *   The GUI subscribes to the `browser_state` Redis channel to display real-time browser status (screenshot, elements, etc.).
        *   Allows sending plain English commands to the `Controller`'s `text_to_browser_action` LLM logic.

**Communication & Message Handling (`busses`, `task_managers`)**

6.  **`busses/bus_manager.py`**:
    *   **Purpose (Original)**: Central communication hub using Python `queue.Queue` and `asyncio.Queue`. It instantiated and started `TaskManager`, `Planner`, `Controller`.
    *   **Purpose (Current Context with Redis)**: While the direct queue management is superseded by Redis, this file's structure shows the intended components and their conceptual linkage. The logging wrappers (`_wrap_sync_method`, `_wrap_async_method`) would have been useful for debugging queue interactions if Python queues were still in use; with Redis, the logging would happen around `redis_client.publish/subscribe`.
    *   **Functionality**:
        *   Defines methods for creating and starting `TaskManager`, `Planner`, `Controller`. If `with_browser_use=True`, it would start `BrowserAssistant` and `BrowserController` (from `off_the_shelf.py`). These are now independent processes communicating via Redis.

7.  **`task_managers/task_manager.py`**:
    *   **Purpose**: Listens to conversation transcripts and identifies user requests that should become executable browser tasks.
    *   **Functionality**:
        *   Subscribes to the `transcript` Redis channel.
        *   Receives messages (conversation history).
        *   Uses an LLM (`_task_organizer_client` with `o4-mini@openai`) with prompts from `task_managers/sys_msgs.py` (e.g., `FIRST_TASK`) and a structured Pydantic response format (`FirstTaskResponse`) to determine if a new task is requested.
        *   If a new task is detected and `first_task.should_create=True`, it logs task details to the Unify "Tasks" context and **publishes** a JSON string containing `[serialized_task_log, task_description]` to the **`text_task`** Redis channel for the `Planner`.

8.  **`task_managers/sys_msgs.py`**:
    *   **Purpose**: Contains system prompt templates for the `TaskManager`'s LLM.
    *   **Functionality**: Defines `FIRST_TASK` and `REORGANISE_TASKS` (for more complex task list management, not fully implemented in `task_manager.py`) prompts to guide the LLM in identifying and structuring tasks from conversation transcripts.

**Planning Layer (`planner`)**

9.  **`planner/planner.py`**:
    *   **Purpose**: Receives high-level task descriptions and (conceptually) decomposes them into lower-level actions. In the current Redis-based version, it's simplified.
    *   **Functionality (Redis version)**:
        *   Subscribes to the `text_task` Redis channel.
        *   Receives a `text_task_log` (Unify Log object as JSON) and `task_description`.
        *   **Currently performs a trivial pass-through**: It treats the `task_description` as a single `text_action`.
        *   **Publishes** this `text_action` to the **`text_action`** Redis channel.
        *   Immediately **publishes** the original `task_description` to the **`task_completion`** Redis channel (signifying task completion prematurely in this simplified version).

10. **`planner/planner_proposal.py`**:
    *   **Purpose**: A more detailed, proposed design for a sophisticated `Planner`. This is not the currently active planner but represents a future direction.
    *   **Functionality (Proposed)**:
        *   `_first_pass`: Initial plan generation using LLMs and cached functions (likely from `FunctionManager`).
        *   `_verify_completes`: Decorator to verify function execution success using LLMs and browser state.
        *   `redirect`: Handles mid-task user commands by modifying the plan and call stack.
        *   This proposal outlines a much more complex planner that would generate and execute Python code representing the plan, involving primitives, verification, and dynamic updates.

**Controller Layer (`controller`)**

11. **`controller/controller.py`**:
    *   **Purpose**: Translates low-level text actions from the `Planner` into specific, executable browser command strings for the `BrowserWorker`, using an LLM informed by the current browser state.
    *   **Functionality**:
        *   Subscribes to the `text_action` Redis channel to get `text_action`.
        *   Subscribes to the `browser_state` Redis channel to get `browser_state` (screenshot, elements, URL, etc.).
        *   Calls `agent.text_to_browser_action` with `text_action` and `browser_state`. This function uses an LLM.
        *   **Publishes** the chosen specific command string (e.g., "click\_button\_3\_login") to the **`browser_command`** Redis channel (note: current code publishes to `browser_state`, which seems like a typo; it should be `browser_command`).
        *   **Immediately publishes** the *original text\_action* back to the **`action_completion`** Redis channel as an acknowledgment to the `Planner`.
        *   Manages starting/stopping the `BrowserWorker`.

12. **`controller/agent.py`**:
    *   **Purpose**: Contains the LLM logic for the `Controller` to map a general text action + browser context to a specific command string.
    *   **Functionality**:
        *   `text_to_browser_action`:
            *   Takes text action, screenshot, tabs, elements, history, browser state.
            *   Uses `action_filter.get_valid_actions` to get possible command prototypes.
            *   (Simple Mode - `ADVANCED_MODE=False`): Prompts an LLM (`o4-mini@openai`) with valid action prototypes and asks it to choose one (and provide a value if needed), using `SimpleChoice` Pydantic model for the response.
            *   (Advanced Mode - `ADVANCED_MODE=True`, not default): More complex Pydantic schema generation (`_create_full_response_format`, `_build_pruned_response_format`) for LLM interaction, allowing for more structured and multi-action responses.
            *   Returns the chosen action details, which `Controller` then formats into a command string.

13. **`controller/action_filter.py`**:
    *   **Purpose**: Determines the set of valid low-level browser action *prototypes* (e.g., `"scroll_down *"`) based on the current `BrowserState`.
    *   **Functionality**: `get_valid_actions` checks `state.in_textbox`, `state.auto_scroll`, dialog state, CAPTCHA state, etc., and returns applicable command strings from `controller.commands`. It can return schemas or action strings.

14. **`controller/commands.py`**:
    *   **Purpose**: Defines canonical string constants and patterns for all low-level browser commands (the vocabulary between `Controller` and `BrowserWorker`).
    *   **Functionality**: Constants like `CMD_SCROLL_DOWN = "scroll_down *"`, `CMD_CLICK_BUTTON = "click_button *"`, etc., and groups of commands (e.g., `TEXTBOX_COMMANDS`).

15. **`controller/states.py`**:
    *   **Purpose**: Defines `dataclass` structures for browser action records (`ActionRecord`), browser state (`BrowserState`), and action history (`ActionHistory`). `BrowserState` includes flags for `dialog_open`, `captcha_pending`.

16. **`controller/gui.py`**:
    *   **Purpose**: Implements a Tkinter-based GUI (`ControlPanel`) for interacting with the browser automation components, primarily for testing and debugging.
    *   **Functionality**:
        *   Displays browser state (elements, tabs, screenshot preview, history, raw state dict).
        *   Allows manual input of commands (search, URL, text entry, key presses via buttons).
        *   Sends these commands to the `browser_command` Redis channel (via `gui_to_browser_queue` in `test_controller.py`).
        *   Receives updates from the `browser_state` Redis channel to refresh its display.
        *   Includes an "LLM Command" input field that sends text to `controller.agent.text_to_browser_action` and then queues the resulting specific command.
        *   Shows/hides a "Solving CAPTCHA..." label based on `state.captcha_pending`.

17. **`controller/captcha_solver.py`**:
    *   **Purpose**: Wraps the `anticaptchaofficial` library to solve different types of CAPTCHAs (reCAPTCHA v2, hCaptcha, image).
    *   **Functionality**:
        *   Reads `ANTI_CAPTCHA_KEY` from `constants.py`.
        *   `solve_recaptcha`, `solve_hcaptcha`, `solve_image`: Functions to call the Anti-Captcha service for specific CAPTCHA types.
        *   `solve`: A generic dispatcher that takes a CAPTCHA payload (as detected by `browser_utils.detect_captcha`) and calls the appropriate solver.
        *   Includes a list of `UNSUPPORTED_DOMAINS` (like google.com) where Anti-Captcha might not work or be advisable.

18. **`controller/sys_msgs.py`**:
    *   **Purpose**: Contains system prompts for the `Controller`'s LLM (`agent.py`).
    *   **Functionality**:
        *   `PRIMITIVE_TO_BROWSER_ACTION_CANDIDATES`: Prompt for when `ADVANCED_MODE` is True, asking the LLM to select multiple candidate actions.
        *   `PRIMITIVE_TO_BROWSER_ACTION`: Prompt for when `ADVANCED_MODE` is True and multiple candidates were selected, asking the LLM to pick the single best one. (The simple mode in `agent.py` uses a different, simpler prompt construction).

**Browser Execution Layer (`controller/playwright`)**

19. **`controller/playwright/worker.py`**:
    *   **Purpose**: The `BrowserWorker` thread runs the Playwright browser instance and executes specific commands received from the `Controller`. It also handles CAPTCHA detection and solving.
    *   **Functionality**:
        *   Runs Playwright in its thread (`sync_playwright`).
        *   Subscribes to `browser_command` Redis channel.
        *   Uses `CommandRunner` to execute commands.
        *   Periodically (and after commands) collects browser state (elements, URL, title, scroll, DOM hash, screenshot via `MirrorPage`).
        *   **CAPTCHA Handling**:
            *   Calls `detect_captcha` (from `browser_utils.py`) to check for CAPTCHAs.
            *   If a CAPTCHA is detected, sets `self.runner.state.captcha_pending = True`.
            *   Spawns a new thread (`_solve_captcha`) to call `captcha_solver.solve` non-blockingly.
            *   Receives solved tokens via `_captcha_q` (a Python queue) and injects them using `_inject_captcha`.
        *   Updates a visual overlay on the browser page using `paint_overlay`.
        *   Publishes the comprehensive state payload (as JSON) to the `browser_state` Redis channel.

20. **`controller/playwright/command_runner.py`**:
    *   **Purpose**: Parses specific command strings and executes corresponding Playwright `Page` API calls.
    *   **Functionality**: Maps command strings from `controller.commands` to Playwright actions (e.g., `page.goto`, `element_handle.click`). Manages active page, dialogs, popups, and updates `BrowserState`. Handles keyboard shortcuts and text input.

21. **`controller/playwright/browser_utils.py`**:
    *   **Purpose**: Playwright utility functions: finding clickable elements (`collect_elements`), formatting data for overlay (`build_boxes`), drawing overlay (`paint_overlay`), launching browser (`launch_persistent`).
    *   **CAPTCHA Detection**: `detect_captcha` function identifies reCAPTCHA v2, hCaptcha, and image CAPTCHAs by inspecting iframe sources and image attributes. It returns a dictionary with `type` and `sitekey` (or `handle` for images).

22. **`controller/playwright/js_snippets.py`**:
    *   **Purpose**: Stores JavaScript code snippets executed in the browser context (e.g., getting element details, drawing overlay, handling scroll).

23. **`controller/playwright/mirror.py`**:
    *   **Purpose**: Manages a headless browser page mirroring the visible page, primarily for taking screenshots without UI overlay interference. It syncs URL, viewport, and scroll position.

**Off-the-Shelf Browser Automation (`off_the_shelf.py`)**

24. **`off_the_shelf.py`**:
    *   **Purpose**: Integrates the `browser_use` library for a more self-contained browser automation solution, as an alternative to the custom `Planner`/`Controller`/`BrowserWorker` stack. This is likely used by `new_make_call.py`.
    *   **Functionality**:
        *   `BrowserAssistant`: A thread that runs a `browser_use.Agent`.
            *   It listens to the `transcript` Redis channel (expecting a JSON with `{"action": "create_task", "payload": messages}`).
            *   When a "create\_task" message arrives, it adds the `payload` (conversation history) as a new task to its `BrowserAgent` instance.
            *   It runs the `BrowserAgent` in a separate asyncio loop (`_browser_loop`) managed by a dedicated thread (`_browser_thread`).
            *   Task results (from `BrowserAgent.run`) are published to the `task_completion` Redis channel.
            *   Uses `State` dataclass (backed by Redis) to manage `task_running`, `task_paused`.
        *   `BrowserController`: A thread that listens to a `command` Redis channel for control messages like "pause\_task", "resume\_task", "cancel\_task" and applies them to the `BrowserAgent` running in `BrowserAssistant`.

**State Management (`state.py`)**

25. **`state.py`**:
    *   **Purpose**: A simple dataclass (`State`) that acts as a façade to Redis for storing and retrieving shared state variables like `task_running`, `task_paused`, `last_task_result`, and `last_step_results`.
    *   **Functionality**: Uses a Redis client to `set` and `get` (with JSON serialization/deserialization) these state flags. This allows different processes (e.g., `VoiceAssistant` in `make_call.py` and `BrowserAssistant` in `off_the_shelf.py`) to share and react to task status.

**Core Managers (as per `communication/sys.md` and module structure)**

26. **`communication/coms_manager.py`**:
    *   **Purpose (Intended)**: Manages all text-based and voice-based communication with the user and potentially other entities. Acts as the central router for incoming messages and outgoing communications. This is a key component for the "Ivy" vision.
    *   **Functionality (Partially Implemented/Conceptual)**:
        *   Initializes `TranscriptManager` and `KnowledgeManager`.
        *   Defines tools for its LLM: `_transcript_manager.ask`, `_knowledge_manager.store`, `_knowledge_manager.retrieve`, and placeholder outbound communication tools (`_send_email`, `_send_sms_message`, `_send_whatsapp_message`, `_make_phone_call`, `_make_whatsapp_call`).
        *   `_receive_email`, `_receive_sms_message`, `_receive_whatsapp_message`: Methods to process incoming messages. They use `_maybe_create_new_contact` (which interacts with `TranscriptManager`) and then call `tool_use_loop` with the `ComsManager`'s LLM and tools.
        *   The `_shift_context_window` decorator resets the LLM's message history to the latest 50 messages from `TranscriptManager` after each interaction.
        *   Uses `o4-mini@openai` as its LLM.

27. **`communication/transcript_manager/transcript_manager.py`**:
    *   **Purpose**: Manages storage, search, and summarization of conversation transcripts.
    *   **Functionality**:
        *   `ask`: Takes a natural language question and uses an LLM (`o4-mini@openai`) with tools (`_search_contacts`, `_search_messages`, `_search_summaries`, `_nearest_messages`) to answer questions about past communications.
        *   `summarize`: Takes `exchange_ids` and uses an LLM to summarize them, storing the result in the "TranscriptSummaries" Unify context.
        *   `log_messages`: Persists `Message` objects to the "Transcripts" Unify context.
        *   `create_contact`, `update_contact`: Methods to persist/update `Contact` data to the "Contacts" Unify context. Handles uniqueness and auto-increments `contact_id`.
        *   `_search_contacts`, `_search_messages`, `_search_summaries`: Internal methods to query Unify logs based on filters.
        *   `_nearest_messages`, `_nearest_summaries`: Internal methods for semantic search using vector embeddings (via `common/embed_utils.py`). `_VEC_MSG` and `_VEC_SUM` define embedding column names.

28. **`communication/transcript_manager/sys_msgs.py`**:
    *   **Purpose**: System prompts for the `TranscriptManager`'s LLM, guiding it on how to use its tools to answer questions about transcripts (`ANSWER` prompt) and how to summarize exchanges (`SUMMARIZE` prompt).

29. **`communication/user_manager.py`**:
    *   **Purpose**: Appears to be an alternative or earlier conceptualization for managing user interactions across different communication channels, similar to `ComsManager` but with a slightly different structure and a more direct use of `sys.md`.
    *   **Functionality**:
        *   Defines `Event` types like `MessageEvent`, `PhoneCallStartedEvent`.
        *   `UserManager` class:
            *   Takes a `Messenger` (e.g., `FileMessenger` for logging to files, `DummyMessenger` for printing to console).
            *   Maintains `past_events` and `current_tasks_status`.
            *   `run()`: Processes `new_events` by creating a prompt based on `communication/sys.md`, calling an LLM (`unify.Unify`) with tools (send\_whatsapp, send\_email etc. via the `Messenger`), and then executing the chosen tool.
        *   This seems more focused on rule-based interaction driven by the `sys.md` prompt.

30. **`communication/sys.md`**:
    *   **Purpose**: A detailed system prompt defining the behavior of a high-level AI assistant (like the one conceptualized in `UserManager` or intended for `ComsManager`).
    *   **Content**: Outlines core responsibilities (conversation, platform switching, task delegation), information sources (events log, task status), communication protocols across different platforms (WhatsApp, Telegram, SMS, Email), task delegation framework, and user interaction guidelines (e.g., maintaining the illusion of a single assistant). This document is **critical** for understanding the *intended* overall system behavior and the "Ivy" demo.

31. **`communication/voice_user_agent/*`**:
    *   **Purpose**: Contains experimental code for a more advanced voice user agent, likely an evolution of `make_call.py`'s `VoiceAssistant`.
    *   **`events.py`**: Defines various event types (e.g., `WhatsappMessageSentEvent`, `PhoneUtteranceEvent`, `InterruptEvent`) with string formatting for logs.
    *   **`actions.py`, `actions_2.py`**: Pydantic models for structured LLM outputs, defining actions like sending messages on different platforms. `actions_2.py` uses a discriminated union for `ActionModel`, which is generally a better practice.
    *   **`sys2.md`**: A system prompt similar to `communication/sys.md` but tailored for the voice agent, emphasizing real-time call handling and function calls for sending messages.
    *   **`voice_user_manager_demo.py`**:
        *   An advanced LiveKit agent (`Assistant`) that processes events from an `asyncio.Queue` (`events_queue`).
        *   `events_aggregator`: A coroutine that collects events, and if an urgent event (like a user utterance) arrives, it cancels any ongoing LLM task and starts a new one.
        *   The `Assistant`'s `llm_node` constructs a prompt from `PAST_EVENTS` and `INFLIGHT_EVENTS` and uses `gpt-4o` with structured output (`AssistantOutput` from `actions_2.py`).

## End-to-End Flow (Voice Mode - Browser Task via `make_call.py` + Redis)

This flow traces a user voice command like "Okay, click the sign-in button" using the `make_call.py` entry point and the custom `TaskManager` -> `Planner` -> `Controller` -> `BrowserWorker` stack communicating via Redis channels. (The `off_the_shelf.py` components would follow a similar pattern but use `BrowserAssistant` which internally uses `browser_use.Agent`).

**0. User Speaks & STT (User -> LiveKit -> VoiceAssistant in `make_call.py`)**
    *   User says: "Okay, click the sign-in button."
    *   LiveKit captures audio, sends it to the Deepgram STT plugin.
    *   STT transcribes the audio to text: "Okay, click the sign-in button."
    *   The `VoiceAssistant.on_user_turn_completed` hook in `make_call.py` is triggered with the new message.
    *   `VoiceAssistant` updates `self._latest_dialogue_window` with the full conversation history including the new user message.
    *   The LLM inside `VoiceAssistant` (from `AgentSession`) decides which tool to call. Assuming it's a browser task and no task is running, it calls its `create_task` tool.
    *   `VoiceAssistant.create_task()`:
        *   Constructs a command: `json.dumps({"action": "create_task", "payload": self._latest_dialogue_window})`.
        *   **Publishes** this JSON string to the **`transcript`** Redis channel.
            *   `self._redis_client.publish("transcript", cmd)`
        *   Returns a confirmation message to the user (e.g., "Alright, let me get on with that.").

**1. Task Identification (TaskManager - `task_managers/task_manager.py`)**
    *   `TaskManager` (running as a separate process) is subscribed to the **`transcript`** Redis channel.
    *   It receives the JSON string, `json.loads` it.
    *   It extracts the `payload` (the list of message dicts).
    *   It calls `self._maybe_update_tasks()` with these messages.
    *   `_maybe_update_tasks()`:
        *   Uses its LLM (`_task_organizer_client` - `o4-mini@openai`) with the `FIRST_TASK` prompt (from `task_managers/sys_msgs.py`) and the received transcript to determine if a task is being requested.
        *   The LLM responds in the `FirstTaskResponse` Pydantic format.
        *   Assuming the LLM identifies "Click the sign-in button" as a new task (`task_was_requested=True`, `first_task.should_create=True`):
            *   `TaskManager` creates a log in the Unify "Tasks" context with details like title ("Click the sign-in button"), description ("Click the sign-in button"), status ("in progress").
            *   It then **publishes** a JSON string `[self._task_log.to_json(), first_task.description]` to the **`text_task`** Redis channel.
                *   `self._redis_client.publish("text_task", json.dumps([self._task_log.to_json(), first_task.description]))`

**2. Planning (Planner - `planner/planner.py`)**
    *   `Planner` (separate process) is subscribed to the **`text_task`** Redis channel.
    *   It receives `[text_task_log_json, task_description]`. It deserializes `text_task_log_json` into a `unify.Log` object.
    *   **Simplified Planner Logic (as per `planner/planner.py`)**:
        *   It directly treats `task_description` ("Click the sign-in button") as the `text_action`.
        *   It **publishes** this `text_action` (as a string/bytes) to the **`text_action`** Redis channel.
            *   `self._redis_client.publish("text_action", text_action)`
        *   It then **publishes** the `task_description` to the **`task_completion`** Redis channel, prematurely signaling completion in this simplified flow.
            *   `self._redis_client.publish("task_completion", task_description)`
    *   *(If a more complex planner from `planner_proposal.py` were used, it would generate a Python plan, call primitives, and wait for `action_completion` for each primitive before sending `task_completion`.)*

**3. Action Conversion & Dispatch (Controller - `controller/controller.py`)**
    *   `Controller` (separate process) is subscribed to the **`text_action`** Redis channel.
    *   It receives the `text_action` (e.g., "Click the sign-in button").
    *   It is also subscribed to the **`browser_state`** Redis channel and gets the latest browser state by calling `self._pubsub_browser_state.get_message()` (this is a blocking get, assuming messages are frequent enough or it waits).
    *   It calls `agent.text_to_browser_action()`, passing the `text_action`, screenshot, tabs, buttons (elements), history, and state from `browser_state`.
        *   Inside `text_to_browser_action` (in `agent.py`):
            *   `get_valid_actions()` is called to determine possible command prototypes.
            *   An LLM (`o4-mini@openai`) is prompted (using simple mode prompt) with the `text_action` and the list of valid actions.
            *   The LLM responds in `SimpleChoice` format, e.g., `{"action": "click_button_*", "value": "sign-in button"}`.
    *   The `Controller` receives this `cmd` dict.
    *   It formats this into a specific command string. For `click_button_*`, it might be `"click button sign-in button"` (the `BrowserWorker` will resolve the element index).
    *   `Controller` **publishes** this specific command string (e.g., `"click button sign-in button"`) to the **`browser_command`** Redis channel. (The code currently says `self._redis_client.publish("browser_state", action)` which should be `browser_command`).
    *   `Controller` **immediately publishes** the *original* `text_action` it received (e.g., "Click the sign-in button") to the **`action_completion`** Redis channel. This signals to the Planner that the action has been dispatched.
        *   `self._redis_client.publish("action_completion", text_action)`

**4. Browser Execution (BrowserWorker - `controller/playwright/worker.py`)**
    *   `BrowserWorker` (separate process/thread) is subscribed to the **`browser_command`** Redis channel.
    *   It receives the specific command string (e.g., `"click button sign-in button"`).
    *   Its `CommandRunner` (`self.runner`) parses this command:
        *   For `"click button sign-in button"`:
            *   It uses `collect_elements()` to get currently visible elements.
            *   It tries to match "sign-in button" (case-insensitive) against element labels from `last_elements`.
            *   If a match is found (e.g., element #3 is "Sign-In Button"), it gets the `ElementHandle`.
            *   It calls `element_handle.click()`.
            *   It updates `self.runner.state.in_textbox` based on the clicked element.
    *   Playwright executes the click action in the browser.
    *   After the action (and periodically):
        *   `BrowserWorker` calls `collect_elements()` again to get `last_elements`.
        *   It calls `paint_overlay()` to draw boxes on the visible page.
        *   It updates `self.runner.state` with current URL, title, scroll position.
        *   It takes a screenshot using `mirror.screenshot()`.
        *   **CAPTCHA Check**: Calls `detect_captcha()`. If a CAPTCHA is found:
            *   Sets `self.runner.state.captcha_pending = True`.
            *   Starts `self._solve_captcha` in a new thread. This thread calls `captcha_solver.solve()`.
            *   The result (token or None) is put onto `self._captcha_q`.
            *   In the main loop, if `_captcha_q` has an item, `self._inject_captcha()` is called to inject the token via JavaScript, and `captcha_pending` is reset.
        *   It **publishes** a comprehensive state payload (elements, tabs, screenshot, history, state dict) as a JSON string to the **`browser_state`** Redis channel.
            *   `self._redis_client.publish("browser_state", json.dumps(payload))`

**5. Feedback to Planner (via Controller's early ACK)**
    *   The `Planner` (if it were complex and waiting) would be subscribed to the **`action_completion`** channel.
    *   It receives the `text_action` ("Click the sign-in button") that the `Controller` published. This would unblock the `Planner` to proceed with the next primitive in its plan (if any). Since the current planner is simple, this message might be ignored or logged.

**6. Task Completion Signal & TTS (Planner -> Speech Dispatcher in `make_call.py`)**
    *   In the **simplified `Planner`**, it already published to `task_completion` in step 2.
    *   The `_speech_dispatcher` logic in `make_call.py` (main async function) is subscribed to the **`task_completion`** channel.
    *   It receives the message (e.g., "Click the sign-in button" from the simplified planner).
    *   It calls `session.interrupt()` to stop any previous TTS.
    *   It calls `session.say("Done. Anything else?")`, triggering the Cartesia TTS plugin.
    *   LiveKit sends the audio to the user.

This flow shows how components communicate asynchronously using Redis pub/sub. The `Controller` provides a quick acknowledgment (`action_completion`) to the `Planner`, while the actual browser state updates (`browser_state`) provide richer context for subsequent decisions. The CAPTCHA solving is neatly integrated into the `BrowserWorker`'s loop.

**Explanation of Architecture Diagram & "Ivy" Demo Mapping:**

1.  **User Interface Layer**: The user interacts via various channels. For "Ivy," this starts with a Phone Call (LiveKit) and involves WhatsApp.
2.  **LiveKit Integration**: For voice, LiveKit handles STT, TTS, WebRTC.
3.  **Central AI Orchestrator (`VoiceAssistant`/`ComsManager`)**:
    *   This is the brain, as described in `communication/sys.md`. It receives user input.
    *   **"Ivy" - Initial Call**: The `VoiceAssistant` (or a future `ComsManager`) handles the initial conversation.
    *   It decides actions:
        *   Direct reply: "While ChatGPT is great..."
        *   Delegate to Specialized Managers (as tools):
            *   **"Ivy" - Sharing Letter Photo**: User sends photo on WhatsApp. `ComsManager` would receive this. A multi-modal LLM tool (not explicitly in current code, but implied for "Ivy") would parse the letter.
            *   **"Ivy" - Storing Utility Info**: `ComsManager` calls `KnowledgeManager.store("Utility Ltd info...")`.
            *   **"Ivy" - Remembering Contact Preferences**: `ComsManager` calls `KnowledgeManager.store("Contact preferences: WhatsApp on, Call on")`.
        *   Delegate to Browser Automation Stack:
            *   **"Ivy" - Resolve Utility Bill**: `ComsManager` decides this is a task. It might first use `TaskListManager.update("Add task: Resolve Utility Bill")`. Then, if the user says "call them tomorrow," it would interact with the browser stack if calling involves a web portal. If it's a direct phone call via a telephony API, `ComsManager` might have a `_make_phone_call` tool. The current code would publish the task description "Call Utility Ltd. at 0800 500 300 to resolve bill" to the `transcript` channel if `VoiceAssistant.create_task` is invoked.
4.  **Redis Pub/Sub Channels**: For browser automation, these are key.
5.  **Browser Automation Stack (`TaskManager` -> `Planner` -> `Controller` -> `BrowserWorker`)**:
    *   If "Call Utility Ltd." involves navigating their website, this stack would be used.
    *   **CAPTCHA**: If the utility website has a CAPTCHA, `BrowserWorker` detects it, `captcha_solver` attempts to solve it, and the token is injected.
6.  **Alternative Browser Flow (`off_the_shelf.py` / `new_make_call.py`)**:
    *   If `new_make_call.py` is used, its `VoiceAssistant` would directly use `BrowserAgent` (from `browser_use` lib, likely via `BrowserAssistant` in `off_the_shelf.py`) for the browser task. `BrowserAssistant` publishes to `task_completion`.
7.  **Specialized Managers**:
    *   `TranscriptManager`: Used by `ComsManager` to recall past interactions ("As you mentioned on Telegram...").
    *   `KnowledgeManager`: Crucial for "Ivy" to store and recall facts about Utility Ltd., contact preferences.
    *   `TaskListManager`: To manage the "Resolve Utility Bill" task's lifecycle.
    *   `FunctionManager`: A more advanced `Planner` could use this to find reusable Python functions for complex tasks.
8.  **"Ivy" - Second Call & Handoff**:
    *   WhatsApp message from Ivy ("Hey Dan, I’m on the call now..."): This would be `ComsManager` (acting as Ivy) sending a message via its `_send_whatsapp_message` tool, after potentially being triggered by an internal state change or a scheduled action from `TaskListManager`.
    *   User replies on WhatsApp. `ComsManager` receives it.
    *   Ivy calls Dan: `ComsManager` initiates this via a `_make_phone_call` tool.
    *   Connecting Dan to Utility Ltd.: This implies advanced call control features, potentially managed by `ComsManager` if it has deep LiveKit integration or a telephony API. The phrase "I can just connect you through directly" suggests call bridging.
    *   Ivy taking over: "Yes, I’m here. Thanks Dan, I can take it from here." This is Ivy (the `ComsManager` / `VoiceAssistant`) rejoining the bridged call or taking back control of the primary call leg.
9.  **"Ivy" - Final Notifications**:
    *   Email confirmation: `ComsManager`'s `_send_email` tool.
    *   WhatsApp "All done 👍": `ComsManager`'s `_send_whatsapp_message` tool. These would be triggered by a `task_completion` signal for the "Resolve Utility Bill" task.

The current codebase has a strong foundation for the browser automation parts (including CAPTCHA solving) and the individual "manager" components. The `communication/sys.md` and the more recent `communication/voice_user_agent/` experiments clearly show the direction towards the full "Ivy" assistant, where `ComsManager` (or an equivalent central orchestrator) uses these managers as tools to fulfill complex, multi-modal, and multi-turn user requests. The Redis architecture supports this by decoupling the browser automation components.

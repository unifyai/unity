**Unify Python Client: Knowledge Base**

`unify` is a Python client library designed to simplify interactions with the Unify REST API. It provides tools for two primary purposes:
1.  **Universal LLM Access (`universal_api/`)**: Interacting with a wide range of Large Language Models (LLMs) through a unified interface, which itself is built on top of LiteLLM but managed via Unify's infrastructure.
2.  **Advanced Logging & Experiment Tracking (`logging/`)**: Recording detailed telemetry, data, parameters, and code context for LLM calls and associated workflows, facilitating debugging, analysis, and experiment management.

Key features like your Unify registered `endpoints`, `credits`, query history & custom endpoint/key creations can also be programmatically managed.

Here's a breakdown of its main components:

**1. Top-Level `unify` Namespace (`unify/__init__.py`)**

*   **Entry Point**: This is how users import and use the library (e.g., `import unify`, `unify.Unify(...)`, `unify.log(...)`).
*   **Configuration**:
    *   `BASE_URL`: Defines the Unify API server address (defaults to `https://api.unify.ai/v0` but can be overridden by `UNIFY_BASE_URL` environment variable).
    *   `UNIFY_KEY`: An environment variable that should hold your Unify API key. This key is automatically read and used for authentication unless an `api_key` is explicitly passed to functions/clients.
    *   `PROJECT`: A global variable that allows you to activate a Unify "Project". Logs and other specific data can be scoped within these projects.
        *   `unify.activate(project_name)`: Sets the active project for subsequent API calls.
        *   `unify.Project(project_name)`: A context manager for temporarily activating a project.
    *   `SEED`: Global seed for reproducible LLM generations.
    *   `LOCAL_MODELS`: A dictionary to register and use local, non-Unify-hosted models as if they were Unify endpoints (e.g., for local testing or custom model functions). `unify.register_local_model(...)` adds models here.
*   **Exports**: Re-exports important classes and functions from submodules, making them directly accessible (e.g., `unify.Unify`, `unify.log`, `unify.list_endpoints`).
*   **Caching**: Initializes access to caching functions, configurable via `unify.set_caching(True/False)` and `unify.set_caching_fname()`.

**2. Universal API (`unify/universal_api/`)**

This submodule is your primary interface for interacting with LLMs through the Unify platform.

*   **Clients (`unify/universal_api/clients/`)**:
    *   These are the core objects for making LLM an LLM generate request. They handle common tasks like setting default parameters (temperature, system messages), managing API keys, making requests, and processing responses.
    *   **`_Client` (`base.py`)**: An abstract base class defining the common interface and parameters for all LLM clients. It manages default generation parameters (`temperature`, `max_completion_tokens`, `messages`, `tools`, etc.), API keys, and settings like `stateful` (to maintain conversation history automatically), `return_full_completion`. *This means all client types inherit methods to get/set these parameters.*
    *   **`Unify` / `AsyncUnify` (`uni_llm.py`)**: The primary client for interacting with a *single* Unify LLM endpoint (e.g., "gpt-4@openai", "claude-3-opus@anthropic").
        *   Initialized typically with `endpoint="model_name@provider_name"` and your `api_key`.
        *   The `generate()` method (and `_generate()` internally) is key to making calls. It accepts a user message and other LLM parameters.
        *   Supports synchronous (`Unify`) and asynchronous (`AsyncUnify`) operations.
        *   Integrates with **caching** (`unify/utils/_caching.py`) for both local file-based and potentially upstream caching mechanism from `_get_cache`, `_write_to_cache`. Caching can an read locally, or write + read locally or from Unify (using an "UNIFY_CACHE" context). Caching settings are passed at initialization or during the `generate()` call.
        *   Properties like `input_cost`, `output_cost`, `ttft`, `itl` can be accessed for a client's configured endpoint, by fetching metrics from `unify.get_endpoint_metrics`.
    *   **`MultiUnify` / `AsyncMultiUnify` (`multi_llm.py`)**: A client for interacting with *multiple* LLM endpoints simultaneously for a single prompt or for multiple parallel prompts/endpoints.
        *   Initialized with a list of endpoints.
        *   Internally, it manages multiple `AsyncUnify` client instances, one for each specified endpoint.
        *   Its `generate()` method distributes the request to all configured endpoints (potentially asynchronously) and can return a dictionary of responses keyed by endpoint, or the first endpoint's if N_endpoints=1 etc. There are helper methods for multi_input generation where each sub_llm is queried with a different argument.
        *   Also has synchronous (`MultiUnify`) and asynchronous (`AsyncMultiUnify`) versions.
    *   **`helpers.py`**: Various internal helper functions to validate parts of an endpoint such as the model, base name or provide e.g., determining if an an endpoint is a "meta" provider- (dynamic model choice at runtime like "lowest-cost<..@providers:openai") etc. These helpers are not meant to be top level Unify calls but assist the innerworkings of the Clients.
*   **ChatBot (`universal_api/chatbot.py`)**:
    *   `ChatBot(client)`: A higher-level abstraction that wraps a `Unify` or `MultiUnify` client to provide a simple command-line interactive chat experience.
    *   Manages conversation history and user input/output.
    *   Has a `run()` method to start the chat loop.
*   **Types (`universal_api/types/prompt.py`)**:
    *   `Prompt`: A simple class used internally by clients to encapsulate all components of an LLM request (messages, parameters, etc.) before sending.
*   **Utilities (`universal_api/utils/`)**:
    *   These modules provide functions to interact with various Unify API endpoints for managing your account, models, and queries. **They are typically available via the top-level `unify` namespace** (e.g., `unify.list_endpoints()`, `unify.get_credits()`).
    *   `custom_api_keys.py`: Functions like `unify.create_custom_api_key()`, `unify.list_custom_api_keys()`, `unify.delete_custom_api_key()` to manage your provider specific-API keys stored on the Unify platform.
    *   `custom_endpoints.py`: Functions like `unify.create_custom_endpoint()` and `unify.list_custom_endpoints()` to register and manage your own private LLM endpoints through Unify.
    *   `endpoint_metrics.py`: `unify.get_endpoint_metrics()` retrieves performance metrics (TTFT, ITL, cost) for specific endpoints. `unify.log_endpoint_metric()` allows users to publish metrics for their custom endpoints.
    *   `supported_endpoints.py`: Functions like `unify.list_providers()`, `unify.list_models()`, `unify.list_endpoints()` to discover available models and providers on the Unify platform.
    *   `credits.py`: `unify.get_credits()` to check your Unify account credit balance.
    *   `queries.py`:
        *   `unify.get_queries()`: To fetch historical LLM queries made through your account.
        *   `unify.get_query_tags()`: To get existing tags you've used.
        *   `unify.log_query()`: To manually log arbitrary query/response pairs to Unify's backend (useful for local models or models not called via `Unify` clients initially).
        *   `unify.get_query_metrics()`: To get aggregated metrics derived from your query-set.
*   **Casting (`universal_api/casting.py`)**:
    *   `unify.cast(...)`: Provides utility functions to convert between common types (e.g., a string to a `Prompt` object, or a `ChatCompletion` to a string). This is mainly for a type-conversion/casting pipeline and no important by itself.
*   **Usage (`universal_api/usage.py`)**:
    *   `unify.with_logging(...)`: A decorator function. If you have a local plain Python function that implements an LLM (or any function you want to log calls for), `unify.with_logging(my_model_func, endpoint="my_local_model@local")` wraps it. When the wrapped function is called, `unify.log_query()` is automatically invoked to log the request and response to the Unify platform under the specified "local" endpoint name.

Here are some key high-level takeaways an LLM agent should understand:

1.  **Dual Purpose:** The Unify client serves two **primary** roles:
    *   **Accessing LLMs:** Through `unify.Unify` (for single models) and `unify.MultiUnify` (for multiple models concurrently), you can send prompts and receive LLM responses. This is the "universal API" aspect.
    *   **Comprehensive Logging:** Through `unify.log` (and its related context managers like `unify.Context`, `unify.Experiment`, `unify.Params`, `unify.Entries`, plus decorators like `@unify.log` and `unify.traced`), you can record *everything* about your LLM workflows—inputs, outputs, intermediate variables, parameters, timestamps, code context, and even traces of internal function calls.

2.  **Central `Unify REST API` Abstraction:** The `unify` Python library is a client that sits on **top of its REST API**. Methods such as `unify.list_projects()` (which calls GET `/projects`) give user control + get functionality straight up from the user profile using a specific key `UNFY_KEY`. The client's `unify/ universal_api_clients)` and the methods e.g for listing and CRUD like management `unify/*utils/*`) abstract these API calls into user_friendly methods from the specific modules. Everything revolves around your `UNIFY_KEY`.

3.  **Projects as a Top-Level Organizer:** The idea of an active `unify.PROJECT` is crucial for the `logging/` functionality. Data logged via `unify.log` is typically scoped within a project (and often further by `Contexts`).

4.  **`unify.log` is a powerful tool:**
    *   It's not just for simple print statements; it's for structured data capture.
    *   Through its context manager capabilities (`with unify.Log(): ...`) and decorators (`@unify.log`), it can simplify complex logging scenarios and integrate deeply into Python Ccode. This goes directly into a Unify's projects (`api.unify`) dashboard and logs which becomes `Unify's Datasets`(Dataframes, as summarized well) across all queries (or `Contexts`).

5.  **Automated Telemetry with Decorators:**
    *   `@unify.log`: Modifies a function's AST to automatically log its (non-underscore) arguments, keyword arguments (**kwargs which isn't underscored), and any variables assigned within the function scope (this excludes `trace`-specific output). It then logs this upon function `return`.
    *   `unify.traced(...)`: This decorator is for
fine-grained function tracing within an active log. It records call signatures, inputs, outputs, execution times, errors, call hierarchy (parent/child spans), and LLM-specific usage metrics if the traced function is an LLM call from this specific Python `unify.api` Unify call context, providing full control over specific python execution flow during that code-path (even your `utils/map`). So a `traced` Unify or multi-LLM `Traced(`Client-Instance)`)` calls in `@log` are now fully queryable + accessible `api-level`. That will likely use `log` (`add_log(_)` and `@log(@unc_inst).log(...)` methods of the main ` Log / Dataset Object` and `Project` that manage global `log entries / parameters and write contexts / read options` too on `the cloud via api logs`.

6.  **Core LLM Client Functionality (`universal_api/`):**
    *   Uses OpenAI-like interface for chat completions.
    *   Handles single calls (`unify.Unify`) and multiple simultaneous calls (`unify.MultiUnify`), usually `through /chat... /openai endpoints`from the Unify platform.
    *   Both sync and async versions (`AsyncUnify`, `AsyncMultiUnify`) are available.
    *   Caching (`unify.cached` and client's `cache=` parameter) for requests to optimize and save costs etc if its an identical model. This uses utils/ + REST endpoints specific from Unify to manage your `keys`.

7.  **Modularity and Top-Level Access:** Most key functionalities are exposed directly under the `unify.` namespace (e.g., `unify.Unify`, `unify.log`, `unify.list_endpoints`, `unify.get_credits`, `unify.Dataset`), making it fairly easy to use for many cases (especially as an Agent).
